One rule explains most of it: data.json is the only dataset the analytical pages touch. Everything else — rek_tabs, sodra, sheets — is either narrative metadata for company cards, or raw material shown only on /explore.

2 — What data.json actually contains

Array of objects, one per {brand, year}. These 12 fields are the entire vocabulary of the analytical UI:

identitycompany, brand, city, activities[], risk
moneyrevenue (= turnover), estimatedIncome (= fee revenue), profit
peopleemployees, avgSalary, salaryCosts, nonSalaryCosts
metayear, _provenance (per-field source tag)
Naming trap that bites everywhere: revenue in the JSON is displayed as Turnover, and estimatedIncome is displayed as Revenue. Every chart does this swap — see RankVsMarket.tsx:42.

3 — Derivation layer between raw and UI

Components never read the array directly. Three modules stand between:

data.ts → MarketModelrows, byBrand[brand][year], brands, segments, years, last (max year with revenue), finYears
metrics.tsmarketTotals, cagr, medianSalary (avgSalary>500), rankOf, margin (profit/estimatedIncome, skipped under €50k)
segments.tsSEG_COLORS, segName (LT→EN), segMetricVal / segMetricPct with min-company gates

Selection state lives in the URL via useDashboardParams.ts: yearmktViewcoViewbasismarketcompanies[]off[]segments[]

5 — The Sodra join, done twice

Both the TS app and the Python legacy build perform the same join independently:

rek_tabs.jsoncompany.tabs["Įmonė"] → row "Įmonės kodas"
↓ lookupdata/sodra/<that code>.json
→ company.sodraattached at load time; never on disk

app/explore/page.tsx:22-43 in TypeScript, legacy-src/build_site.py:22-72 in Python. Change the join logic and you must change both.
