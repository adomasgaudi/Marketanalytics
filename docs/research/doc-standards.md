# Research — how to document a repo for another engineer or AI

> Question: should a repo "map" be one dense detailed file, or a short brief linking to detail — and do you include line numbers? Researched 2026-07-11 via the `keyword-research` method (surface expert vocabulary → research named authorities → grade sources → converge).

## Per-source findings

### 1. Diátaxis (Daniele Procida) — **Grade A**

Primary source (diataxis.fr); adopted by Python, Django, Gatsby, Cloudflare.
**Prescription:** a repo map is **reference** — "technical descriptions of the machinery and how to operate it," information-oriented, "austere, factual, neutral." Load-bearing rule: **"the structure of the documentation should mirror the structure of the product."** Reference is _led by the product_ and **must not mix in explanation** (the "why" belongs in a separate _explanation_ doc). So: split the terse structural map from the discursive why. Not one blended document.
Sources: https://diataxis.fr/reference/ · https://diataxis.fr/

### 2. C4 model & arc42 (Simon Brown) — **Grade A**

**Prescription:** four **zoom levels** — Context → Container → Component → Code — separating summary from detail by audience. **Level 4 (Code) is the level teams are told to mostly skip**: it "goes into too much technical detail" and where used should be "ideally autogenerated from the actual code," never hand-maintained. → Keep hand-written code-level detail **out** of the maintained top doc; generate on demand.
Sources: https://www.infoq.com/articles/C4-architecture-model/ · https://c4model.com/faq

### 3. ADRs — Architecture Decision Records (Michael Nygard) — **Grade A**

**Prescription:** the **"why" lives in ADRs, separate from the structural map.** One ADR = one significant decision (context, decision, consequences). **ADRs are immutable** — superseded, not edited — the opposite lifecycle to the current-state map. "No one reads large documents," but losing the rationale is "disastrous." Map answers _what/where_; ADRs answer _why_; don't fuse them.
Sources: https://cognitect.com/blog/2011/11/15/documenting-architecture-decisions · https://docs.arc42.org/examples/decision-use-adrs/

### 4. Living Documentation (Cyrille Martraire) — **Grade A**

Directly on line-number rot. **Prescription:** docs should be "reliable, low-effort, evergreen, refactoring-proof." **Separate stable knowledge from volatile, write down only the stable part.** Anchor to **stable identifiers, not volatile positions**; make code self-documenting (naming, types, annotations); **generate** from source (tests, AST) rather than hand-writing coordinates. **Line numbers are the textbook volatile anchor** — exactly what to avoid. Prefer stable symbol names (class/function/module) that survive refactoring.
Sources: https://www.oreilly.com/library/view/living-documentation-continuous/9780134689418/ · https://ddd.academy/living-documentation-cyrille-martraire/

### 5. LLM / agent context files — **Grade B** (vendor docs + strong community consensus)

- **Aider repomap** — the reference architecture: built **automatically with tree-sitter**, extracts **symbol definitions and full signatures** (dense symbols, _not prose_), ranks identifiers via a **PageRank-style** graph, **regenerates per chat**, stores **signatures, not line numbers** — positional info deliberately absent because volatile.
  https://aider.chat/docs/repomap.html · https://aider.chat/2023/10/22/repomap.html
- **Anthropic CLAUDE.md** — keep it **concise** (~<200 lines), universally-applicable facts only. "Mistake: pasting code — use file references and pointers." Favours a **short brief that links out**, no line numbers.
  https://code.claude.com/docs/en/memory
- **Cursor / Cline** — convergent: small, scoped, modular markdown rules in repo root (Cursor `.cursor/rules/*.mdc` glob-scoped; Cline `.clinerules/` concatenated). None prescribe line numbers.
  https://codersera.com/blog/agents-md-vs-claude-md-vs-cursor-rules-comparison-2026/

## Synthesis / recommendation

The authorities converge hard: **short brief + a structural map keyed to STABLE anchors (symbol/file names), and NO line numbers** — with the "why" and deep symbol detail pushed to separate, appropriately-lifecycled artifacts.

- Diátaxis: split terse _reference_ from _explanation_.
- C4: layer by zoom; keep hand-maintained code detail out.
- Nygard: immutable _why_ lives in ADRs, not the map.
- Martraire: write only the _stable_ part; generate the volatile part.
- Aider: proven pattern — generated dense signatures, **no line numbers**.

**Line-number verdict (unanimous):** never put line numbers in a hand-maintained doc — the canonical volatile anchor, stale on the next edit. Use stable identifiers; if positional precision is truly needed, **generate it on demand**. A single fat blended map is the anti-pattern all five reject.
